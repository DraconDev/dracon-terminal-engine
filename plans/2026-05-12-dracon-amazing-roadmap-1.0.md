# Dracon Terminal Engine — Strategic Roadmap to "Amazing"

## Objective

Identify and prioritize the highest-impact improvements that will differentiate Dracon from other TUI frameworks (ratatui, crossterm, etc.) and fully realize the vision of **GUI-grade terminal applications**.

---

## Analysis Methodology

- Audited all 78K lines across `src/`, `examples/`, and `tests/`
- Examined 37 framework widgets, 21 themes, compositor, layout, animation, input, scene router, event bus, drag-and-drop, plugin system
- Compared against common GUI patterns and competing TUI frameworks
- Evaluated each gap by: **User Impact**, **Architectural Fit**, **Effort**, **Differentiation**

---

## Priority 1 — Critical Gaps (Do These First)

### 1.1 Layout Engine: Add Vertical & 2D Layout

**Problem**: `src/framework/layout.rs:70` is 1D horizontal only. No vertical stacking, no grid, no flex-wrap. Every widget does manual y-coordinate math. This is the #1 friction point for building real apps.

**Impact**: High — every new widget reinvents vertical layout
**Effort**: Medium
**Differentiation**: Medium (ratatui has this; we need parity)

- [ ] Add `Direction::Vertical | Direction::Horizontal` to `Layout`
- [ ] Add `Layout::vertical(constraints)` and `Layout::horizontal(constraints)` constructors
- [ ] Add `Margin` struct and `Layout::margin(margin)` for padding
- [ ] Add `Alignment::Start | Center | End` for distributing leftover space
- [ ] Add `Constraint::Fill` for "take all remaining space" (like flex-grow)
- [ ] Add `Layout::split(area) -> [Rect; N]` convenience for 2-4 children (common case)
- [ ] Update `examples/_apps/file_manager.rs` and others to use new layout instead of manual rect math

### 1.2 Rich Text / Markdown Renderer Widget

**Problem**: No way to render styled inline text. Every widget that needs bold + color + links within a single line manipulates cells manually. No markdown support means help text, chat messages, and documentation are painful.

**Impact**: Very High — chat_client, text_editor, help overlays all need this
**Effort**: Medium-High
**Differentiation**: High (no TUI framework has a great markdown renderer)

- [ ] Create `src/framework/widgets/rich_text.rs` — `RichText` widget
- [ ] Define `TextSpan { text, fg, bg, style, link: Option<String> }`
- [ ] Support word-wrap with proper span boundary handling
- [ ] Support clickable hyperlinks (via HitZone dispatch or OSC 8)
- [ ] Create `MarkdownRenderer` that parses a subset of Markdown into `TextSpan`s
- [ ] Support: headers, bold, italic, inline code, code blocks, links, bullet lists, blockquotes
- [ ] Add syntax highlighting for code blocks via syntect integration
- [ ] Use in `chat_client.rs` for message rendering and `text_editor_demo.rs` for help

### 1.3 Bracketed Paste Support

**Problem**: `src/input/parser.rs` does not handle bracketed paste sequences (`\x1b[200~` ... `\x1b[201~`). Pasting multi-line text into TextEditor or SearchInput inserts escape characters and breaks the UI.

**Impact**: Very High — every text input widget is broken for paste
**Effort**: Low
**Differentiation**: Low (expected baseline behavior)

- [ ] Add `Event::Paste(String)` variant to `src/input/event.rs`
- [ ] Detect bracketed paste start `ESC[200~` in parser, accumulate until `ESC[201~`
- [ ] Enable bracketed paste mode in `Terminal::enter_raw_mode()` via `ESC[?2004h`
- [ ] Disable on exit via `ESC[?2004l`
- [ ] Handle `Event::Paste` in TextEditor, SearchInput, PasswordInput, TextInputBase
- [ ] Add test in `tests/input_parser_tests.rs` for paste sequence decoding

### 1.4 Notification Center / Toast Queue Widget

**Problem**: `Toast` widget exists but is a single static widget. No queue, no auto-dismiss, no stacking. GUI apps have notification centers.

**Impact**: High — every app needs non-blocking user feedback
**Effort**: Medium
**Differentiation**: Medium

- [ ] Create `src/framework/widgets/notification_center.rs`
- [ ] `Notification { id, title, body, severity, duration, timestamp }`
- [ ] Auto-dismiss after duration with optional progress bar
- [ ] Stack vertically with max height, scrollable history
- [ ] Severity icons/colors: info, success, warning, error
- [ ] `NotificationCenter::push(notification)` — adds to queue
- [ ] `NotificationCenter::dismiss(id)` — manual dismiss
- [ ] Integration with `EventBus` so any widget can publish notifications
- [ ] Add to `ide.rs` example for save/copy/paste feedback

## Priority 2 — High Impact, Medium Effort

### 2.1 Form Validation Framework

**Problem**: `Form` widget has `set_field_error()` but no validation rules engine. Cross-field validation, async validation, and standard rules (email, regex, min/max length) are all manual.

**Impact**: High — forms are a core widget, validation is table stakes
**Effort**: Medium
**Differentiation**: Medium

- [ ] Create `src/framework/validation.rs` — rule engine
- [ ] `ValidationRule::Required | MinLength(usize) | MaxLength(usize) | Regex(&str) | Email | Custom(Fn) | And(Vec) | Or(Vec)`
- [ ] `Validator::validate_field(&self, value: &str) -> Result<(), Vec<String>>`
- [ ] `Form::with_validator(field_index, rule)` builder method
- [ ] `Form::validate_all() -> Result<(), HashMap<usize, Vec<String>>>`
- [ ] Real-time validation on blur (when focus leaves field)
- [ ] Cross-field validation via `ValidationRule::Custom` with closure accessing form state
- [ ] Update `form_widget.rs` and `form_demo.rs` examples to show validation

### 2.2 Autocomplete / Typeahead Widget

**Problem**: No dropdown suggestion list for text inputs. SearchInput exists but no autocomplete. Essential for command palettes, file paths, tags.

**Impact**: High — common GUI pattern, currently impossible
**Effort**: Medium
**Differentiation**: Medium

- [ ] Create `src/framework/widgets/autocomplete.rs`
- [ ] `Autocomplete<T>` wraps any text input, shows suggestion dropdown below
- [ ] `Suggestion { text, display_text, data: T }`
- [ ] `.with_suggestions_fn(Fn(&str) -> Vec<Suggestion<T>>)` — sync source
- [ ] `.with_async_suggestions(Fn(&str) -> Future<Vec<Suggestion<T>>>)` — requires `async` feature
- [ ] Keyboard: Up/Down navigate suggestions, Tab accepts, Esc dismisses
- [ ] Mouse: click suggestion to accept, click outside to dismiss
- [ ] Highlight matching substring in suggestions
- [ ] Use in `ide.rs` command palette for fuzzy command matching
- [ ] Use in `text_editor_demo.rs` for file path completion

### 2.3 Calendar / Date Picker Widget

**Problem**: No date selection widget. Common in forms, scheduling apps, log filtering.

**Impact**: Medium-High
**Effort**: Medium
**Differentiation**: Medium

- [ ] Create `src/framework/widgets/calendar.rs`
- [ ] `Calendar` widget showing month grid with navigation
- [ ] `Calendar::with_selected(date)` — pre-select a date
- [ ] `Calendar::with_range(min, max)` — restrict selectable dates
- [ ] Keyboard: arrow keys navigate days, Enter selects, PgUp/PgDn change month
- [ ] Mouse: click day to select, click header arrows for month/year
- [ ] Emit `SelectCallback<NaiveDate>` on selection
- [ ] Add to `form_widget.rs` example as a date field

### 2.4 Plane / Cell Object Pooling

**Problem**: Every `render()` call allocates a new `Plane` with `Vec::with_capacity(width * height)`. For 80x25 terminals that's 2,000 cells per widget per frame. With 10 widgets and 60fps, that's 1.2M allocations/minute.

**Impact**: High — performance bottleneck for complex apps
**Effort**: Medium
**Differentiation**: Low (expected for a quality engine)

- [ ] Create `src/compositor/pool.rs` — `PlanePool`
- [ ] `PlanePool::acquire(width, height) -> Plane` — reuse or allocate
- [ ] `PlanePool::release(plane)` — return to pool, clear cells
- [ ] Thread-local pool per thread to avoid synchronization
- [ ] Cap pool size per dimension to prevent unbounded growth
- [ ] Benchmark before/after with `criterion` in `benches/framework_benchmarks.rs`
- [ ] Target: 50%+ reduction in allocation rate for typical apps
